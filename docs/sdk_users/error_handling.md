# Error Handling

This guide explains the main transaction and query errors raised by the Hiero Python SDK and how to handle them in user applications.

## Overview

Most SDK operations move through three stages:

1. **Precheck:** A node validates the transaction or query before it reaches consensus. Failures here raise `PrecheckError`.
2. **Network retry:** The SDK retries temporary network or node conditions until the request succeeds, times out, or reaches the configured maximum attempts. Exhausted retries raise `MaxAttemptsError`.
3. **Receipt validation:** A transaction can reach consensus but still fail with a non-`SUCCESS` receipt status. When receipt validation is enabled, this raises `ReceiptStatusError`.

## PrecheckError

`PrecheckError` is raised before a transaction reaches consensus.

**When raised:** The transaction or query fails pre-consensus validation at a node.

**Attributes:**

| Attribute | Type | Description |
| --- | --- | --- |
| `status` | `ResponseCode` | Precheck status returned by the node. |
| `transaction_id` | `TransactionId | None` | Transaction ID associated with the failed request, when available. |
| `message` | `str` | Human-readable error message generated by the SDK or supplied by the caller. |

Common status codes include:

- `INSUFFICIENT_TX_FEE`
- `INVALID_SIGNATURE`
- `TRANSACTION_EXPIRED`

```python
from hiero_sdk_python.exceptions import PrecheckError

try:
    response = transaction.execute(client)
except PrecheckError as error:
    print(f"Precheck failed: {error.status.name} ({int(error.status)})")
    print(f"Transaction ID: {error.transaction_id}")
    print(error.message)
```

Fix the transaction inputs before retrying non-transient precheck failures such as `INSUFFICIENT_TX_FEE` or `INVALID_SIGNATURE`.

## MaxAttemptsError

`MaxAttemptsError` is raised when the SDK cannot complete a request within the retry configuration.

**When raised:** All retry attempts are exhausted or the request timeout is exceeded.

**Attributes:**

| Attribute | Type | Description |
| --- | --- | --- |
| `node_id` | `str` | Node account ID used when the SDK stopped retrying. |
| `last_error` | `BaseException | None` | Last underlying error seen by the SDK, when available. |
| `message` | `str` | Error message, including the last error text when present. |

The `last_error` may be a retried `PrecheckError`, such as `BUSY`, or a transport-level gRPC error from the network.

```python
from hiero_sdk_python.exceptions import MaxAttemptsError

try:
    response = transaction.execute(client, wait_for_receipt=False)
    receipt = response.get_receipt(client)
except MaxAttemptsError as error:
    print(f"Retries exhausted on node: {error.node_id}")
    print(error.message)
    if error.last_error is not None:
        print(f"Last error type: {type(error.last_error).__name__}")
        print(f"Last error: {error.last_error}")
```

Manual retry may help when the cause is a temporary network condition, but use backoff rather than immediately looping.

## ReceiptStatusError

`ReceiptStatusError` is raised after consensus when a receipt reports a failure status.

**When raised:** The transaction reached consensus, but the receipt contains a non-`SUCCESS` status and receipt validation is enabled.

**Attributes:**

| Attribute | Type | Description |
| --- | --- | --- |
| `status` | `ResponseCode` | Receipt status returned by the network. |
| `transaction_id` | `TransactionId | None` | Transaction ID associated with the receipt, when available. |
| `transaction_receipt` | `TransactionReceipt` | Receipt object containing the failure status and receipt fields. |
| `message` | `str` | Human-readable error message generated by the SDK or supplied by the caller. |

The `validate_status` parameter controls whether receipt queries raise this exception. When `validate_status=False`, the SDK returns the receipt even if its status is not `SUCCESS`.

```python
from hiero_sdk_python.exceptions import ReceiptStatusError

try:
    receipt = transaction.execute(client, validate_status=True)
except ReceiptStatusError as error:
    print(f"Receipt status: {error.status.name} ({int(error.status)})")
    print(f"Transaction ID: {error.transaction_id}")
    print(error.transaction_receipt)
```

Use receipt details to decide whether the transaction can be corrected and submitted again.

## Understanding ResponseCode

`ResponseCode` is an `IntEnum` with approximately 394 named values from the network protobuf status list.

Key values include:

| Value | Integer | Meaning |
| --- | ---: | --- |
| `OK` | `0` | Node accepted the transaction or query precheck. |
| `BUSY` | `12` | Node is temporarily busy and the SDK may retry. |
| `UNKNOWN` | `21` | Receipt status is not final yet and the SDK may retry receipt lookup. |
| `SUCCESS` | `22` | Transaction reached consensus successfully. |

```python
from hiero_sdk_python.response_code import ResponseCode

status = ResponseCode.BUSY
print(status.name)   # BUSY
print(int(status))   # 12
```

If the network returns a status code not yet defined by the SDK, `ResponseCode` creates a synthetic member such as `UNKNOWN_CODE_999`. Use the `is_unknown` property to detect it.

```python
status = ResponseCode(999)
if status.is_unknown:
    print(f"Unknown response code: {int(status)}")
```

The SDK does not include human-readable descriptions beyond the enum name, so application logs should include both `status.name` and `int(status)`.

## Practical Example

The recommended pattern is to catch the most specific SDK exceptions around the call that submits the transaction and fetches its receipt.

```python
from hiero_sdk_python.exceptions import (
    MaxAttemptsError,
    PrecheckError,
    ReceiptStatusError,
)
from hiero_sdk_python.response_code import ResponseCode

try:
    response = transaction.execute(client, wait_for_receipt=False)
    receipt = response.get_receipt(client, validate_status=True)
    if receipt.status == ResponseCode.SUCCESS:
        print(f"Transaction succeeded: {receipt.transaction_id}")

except PrecheckError as error:
    print(f"Precheck failed: {error.status.name} ({int(error.status)})")
    print(f"Transaction ID: {error.transaction_id}")

except ReceiptStatusError as error:
    print(f"Receipt failed: {error.status.name} ({int(error.status)})")
    print(f"Transaction ID: {error.transaction_id}")
    print(f"Receipt: {error.transaction_receipt}")

except MaxAttemptsError as error:
    print(f"Request did not complete after retries on node {error.node_id}")
    print(error.message)
    if error.last_error is not None:
        print(f"Last error: {error.last_error}")
```

## Retry and Backoff Guidance

The SDK automatically retries transient statuses such as `BUSY`, `PLATFORM_NOT_ACTIVE`, and `PLATFORM_TRANSACTION_NOT_CREATED`. Receipt queries also retry statuses such as `UNKNOWN`, `RECEIPT_NOT_FOUND`, and `RECORD_NOT_FOUND` while the network is still finalizing the transaction.

Do not manually retry immediately for statuses that indicate an invalid request, such as `INSUFFICIENT_TX_FEE` or `INVALID_SIGNATURE`. Fix the fee, signature, account, or transaction fields first.

Manual retry may help after `MaxAttemptsError` when the last error points to temporary network pressure or a short outage. Use exponential backoff with a cap, and consider logging `node_id`, `last_error`, `status.name`, and `int(status)` for diagnosis.
